I am using pure intel syntax x86 assembly, and as such I don’t really have “variables”, but rather a multitude of labels, such as the active disk, the SysFlags boolean collection, VGA_BASE, which is the base segment of the VGA memory for the associated screen, and VAddr, which points to the current video memory address. I also have numerous constants, such as BootLRD and BootLKF, which are informational messages, SCAN, which contains scan code set 2 scan codes, in the order I have in LOWER and UPPER which contain all the characters associated with given standard keys, in their unshifted and shifted forms, respectively. The ScanCodeToChar function iterates over SCAN to locate ah in SCAN. This is done using two cmp(Compare) and je(Jump Equal) instruction pairs, the first pair, which compares the current item of SCAN to zero, breaks the loop as failure if the item has not been found, and the other compares the current item with the contents of ah(Our target), and breaks the loop as success. The end of the loop has jmp .loop(Jump to the start of the loop) in order to loop back to the beginning if this item was not our target. If we did not have our “list”(it is just a bunch of consecutive data addresses, since a list is a much higher level concept), the ScanCodeToChar would be a several kilobyte long section(possibly even a megabyte of assembly). One procedure I developed is ReadInput which is a part of the standard library for Femto, and is intended to read an item off of the INPUT_BUFFER. This is done by pointing to the input buffer, adding the read head to our pointer, reading our data, then moving the read head forward(reset to zero if it exceeds our buffer’s size), and then returning our data in al. Nothing is done if the data is blank. This procedure uses selection by invoking .reset if the read head is passed the end of the buffer(cmp byte [cs:IRHEAD], 32 and je .reset)

ReadInput:
    	push bx
    	mov bx, INPUT_BUFFER
    	add bl, [cs:IRHEAD]
.block:
    	mov al, [cs:bx]
    	cmp al, 0
    	je .block
    	mov byte [cs:bx], 0
    	inc byte [cs:IRHEAD]
    	cmp byte [cs:IRHEAD], 32
    	je .reset
    	pop bx
    	ret
.reset:
    	mov byte [cs:IRHEAD], 0
    	pop bx
    	ret

Code block which calls ReadInput, and handles it’s values.(.prompt is a label, signifying the start of this code block)

.prompt:
    	call ReadInput
    	cmp al, 10
    	je .exec
    	cmp al, 8
    	je .bs
    	mov byte [Char], al
    	mov bx, Char
    	call print
    	jmp .prompt

One procedure I developed with a parameter is print, which iterates through a string of characters, and moves them into the video memory at an increasing address. If it detects a new line, or carriage return it modifies the target address accordingly and continues. At the end of the function, if the address is greater than 4000, it moves all addresses back by 160 (One line), and retries the completion sections. This procedure uses selection in multiple places such as invoking the newline handler(cmp al, 10 and je .newline), the carriage return handler(cmp al, 13 and je .cr), escaping a completed print statement(cmp al, 0 and je.end) and accessing the scroll process(cmp di, 4000 and jg .scroll (The scroll process is sort of a subfunction, but still requires the same state))

print:
    	pusha
    	push es
    	mov ax, [cs:VGA_BASE]
    	mov es, ax
    	mov di, [cs:VAddr]
    	mov dl, [cs:Color]
.loop:
    	mov al, [ds:bx]
    	cmp al, 10
    	je .newline
    	cmp al, 13
    	je .cr
    	cmp al, 0
    	je .end
    	mov byte [es:di], al
    	inc di
    	mov byte [es:di], dl
    	inc di
    	inc bx
    	jmp .loop
.newline:
    	add di, 160
.cr:
    	mov ax, di
    	div byte [cs:COLS]
    	mul byte [cs:COLS]
    	mov di, ax
    	inc bx
    	jmp .loop
.scroll:
    	mov bx, 160
    	push di
    	mov di, 0
.scroll_loop:
    	mov al, [es:bx]
    	inc bx
    	mov [es:di], al
    	inc di
    	cmp di, 4000
    	jle .scroll_loop
    	pop di
    	sub di, 160
.end:
    	cmp di, 4000
    	jg .scroll
    	mov word [cs:VAddr], di
    	pop es
    	popa
    	ret
